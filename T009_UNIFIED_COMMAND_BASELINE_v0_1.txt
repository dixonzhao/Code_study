Status: FORMAL WORKING BASELINE
Date: 2026-07-21
Scope: P001-P004 current system, confirmed behavior, governance, and refactor order

This document is the single working baseline after T001-T008 characterization.
It does not authorize new algorithms. It does not claim root cause.


1. MISSION

The first engineering objective is:

Convert a large camera/radar/CAN/video dataset into a small, reviewable list of high-priority discrepancy candidate windows linked to Framecount/video position.

The current objective is NOT:

- automatic root-cause judgment;
- automatic camera fault judgment;
- automatic radar fault judgment;
- adding every possible statistical or SysID method;
- building new Analyzer modules before the current chain is controlled.

Formal terminology:

- Radar is a provisional engineering reference, not ground truth.
- Reference minus Target is a discrepancy, not automatically camera error.
- Baseline is an estimate of normal conditional discrepancy structure.
- Residual is discrepancy minus the selected baseline.
- Event Hint is a review candidate window, not a confirmed business event.

4. CURRENT ACTUAL EXECUTION CHAIN

Current operational chain:

P001 optional signal discovery
-> user manually selects CAN IDs, Channels, DBCs, and Signals
-> P002 decodes Raw MATLAB CAN CSV
-> user usually selects P002 output in P003
-> P003 selects/cleans/sorts/types/fills output
-> user selects P002 or P003 output in P004
-> user manually maps Frame/Reference/Target/Stage columns and parameters
-> P004 creates discrepancy/residual/investigation output
-> human engineering review checks video/CAN/tracking/HMI context

There is currently:

- no code-level orchestrator;
- no shared schema registry;
- no versioned handoff contract;
- no compatibility test between program versions;
- no common run manifest;
- no automatic program-to-program invocation.

The effective interface is currently:

user memory + GUI configuration + column-name resolution + runtime trial.


5. TIME AND FRAME AUTHORITY

USER-CONFIRMED FACT:

- DateTime is not trustworthy for the current analysis.
- Time[us] is not trustworthy for the current analysis.
- Framecount is the only trusted video/data alignment reference.
- The stereo-camera video visibly provides Framecount for manual verification.

Consequences:

- Future formal sorting/alignment/event contracts must be Framecount-centered.
- Row adjacency is not equivalent to Framecount continuity.
- A program must not silently replace invalid/missing Framecount with zero.
- Framecount Gap, duplicate, invalid, and possible wrap behavior require explicit policy.
- Video-linked outputs must retain start, peak, and end Framecount.

UNRESOLVED BUSINESS CONTRACT:

- exact policy for multiple rows within one Framecount;
- exact policy for duplicate Framecount values;
- whether a future canonical table is event/update-row based or one-row-per-Framecount;
- Framecount wrap/reset behavior across files or video segments.


6. PROGRAM INVENTORY AND CONFIRMED BEHAVIOR


6.1 P001 - Header Catalog

Current valid use:

- optional exploration when available Signals/Headers are unknown;
- human reference for Ctrl-F/search and later P002 configuration.

Confirmed:

- P001 is not programmatically consumed by P002-P004.
- P001 contains physical decode capability beyond the minimum catalog role.
- P001 duplicates more than five normalized parser/decode functions with P002.
- P001 and P002 share a common origin but have meaningful divergence.
- P001 and P002 do not import a shared local parser/decoder module.
- P001-only Unit handling exists in the inspected branch.
- Catalog/decode divergence can theoretically cause P001 and P002 to disagree about supported Signals.

Formal target responsibility:

- discover structure using a shared CAN/DBC core;
- present catalog information;
- do not maintain an independent physical decode implementation.


6.2 P002 - CAN/DBC Decoder

Current valid use:

- authoritative existing Raw CAN -> decoded physical signal implementation.

Runtime-confirmed strengths:

- real P002 decode path executed on Synthetic CSV/DBC;
- Little Endian correct in tested case;
- Big Endian correct in tested case;
- Signed decode correct in tested case;
- Scale and Offset correct in tested case;
- Channel selection correct in tested case;
- Framecount decoded correctly in tested case;
- incomplete/invalid byte updates produced warning/skip behavior;
- output Wide CSV was created;
- tested same-name collision path produced separate/renamed columns rather than silent overwrite.

Confirmed limitations:

- DBC Min/Max are ignored;
- Extended CAN ID is not supported in current P001/P002 inspection;
- Unlimited Forward Fill is applied;
- no Forward Fill disable/limit contract was confirmed;
- no update mask;
- no signal age / last update;
- no original source row ID;
- no formal Unit/Validity metadata propagation;
- multiple rows can exist within one Framecount;
- output naming/collision behavior has mixed branches;
- only partial output schema validation exists.

Formal target responsibility:

- decode bytes to physical signal updates using a shared tested CAN/DBC core;
- preserve raw update identity and provenance;
- not decide final missing-value semantics silently;
- expose data required by the canonical dataset builder.


6.3 P003 - Cleaner

Current operational use:

- usually used because Framecount-based sorting/alignment and analysis-column preparation are required before analysis.

Confirmed behavior:

- supports CSV and Excel;
- reads full file into memory;
- can read P002 output without required filename coupling;
- Framecount is treated as a required/current central column in inspected path;
- stable sort is explicitly requested after numeric sort-key coercion;
- duplicate sort/frame keys are preserved;
- kept Signal column names are preserved;
- can conditionally change row order and dtype;
- no original source row ID is created/preserved by contract;
- no Framecount Gap or Wrap detection;
- no Forward Fill inside P003;
- default remaining Blank/NaN action is Fill Zero;
- Zero Fill covers the generic remaining-column path;
- Framecount can enter the generic Zero Fill path;
- current Zero Fill cannot be disabled through the confirmed path;
- no Signal-specific invalid sentinel contract;
- no distinction among No Update, Invalid, Unavailable, and True Zero;
- no update mask;
- no Signal Age;
- no imputation log or cell-level trace;
- numeric conversion failure becomes NaN and may later become zero;
- cleaning logic is distributed across multiple class methods;
- no independent headless Cleaner Core exists;
- repeated T004C/T008 attempts could not execute real P003 without GUI coupling/refactor.

Critical semantic risk:

P002 Unlimited Forward Fill
-> P003 remaining missing values become zero
-> missingness provenance is lost
-> P004 can interpret imputed values as ordinary measurements.

Formal target responsibility:

- become a Dataset Builder/Validator with a callable headless Core;
- own explicit sorting, validation, optional alignment, and approved imputation policy;
- never convert unknown/invalid/missing Framecount to zero silently;
- preserve source/update/age/validity metadata.


6.4 P004 - Stage Residual Analyzer

Runtime-confirmed strengths:

- imports successfully;
- Reference-Target direction works in tested case;
- Existing Discrepancy mode is consistent with direct discrepancy in tested case;
- Stage1 real implementation executes;
- Stage2 real implementation executes;
- Stage1 and Stage2 Residual outputs matched fixed Synthetic expectations;
- Stage Core is mainly a pure-function/small-connected-function set;
- Stage Core extraction without mathematical change is feasible;
- Pearson result present/directionally correct in tested case;
- Spearman result present/directionally correct in tested case;
- NMI result present/in valid range in tested case;
- Threshold produced expected candidates;
- High-Risk Count Gate was applied in tested path;
- P004 can accept generic P002 output through file/column mapping without a hard-coded P003 filename.

Confirmed limitations:

- Baseline field representation did not match T004E fixed expectation while Residual did; Baseline output contract requires later clarification before declaring a math defect;
- Baseline fit and Stage metrics use the same rows;
- Minimum Sample Count Gate exists but is optional/disabled in confirmed path;
- Sparse single-sample Bin was rejected/invalidated in tested case;
- Lag is DataFrame Row Lag, not Framecount Difference;
- Event Hint continuity is Row-based;
- Candidate Rows across a large Framecount Gap were merged into one Event Hint;
- Event start/end/peak output was only partially validated;
- Invalid/Missing/Unit/Row Identity handling is mixed and configuration-dependent;
- P004 may coerce broad input columns numerically in an inspected path;
- P004 contains Stage analysis, association scan, high-risk analysis, Event Hint generation, and reporting in one expanding program.

Formal target responsibility:

- analyze a validated canonical dataset;
- compute approved discrepancy/baseline/residual methods;
- expose Stage Core separately from GUI/reporting;
- not silently redefine row order or missingness;
- not claim root cause.

Event processing target:

- separate Event Candidate processing responsibility from Stage mathematics;
- use Framecount continuity/gap rules, not only adjacent DataFrame rows;
- output start/peak/end Framecount and review context.


7. FORMAL TECHNICAL DEBT REGISTER

TD-001 CRITICAL - Missingness provenance loss

P002 Forward Fill plus P003 Zero Fill makes stale/missing/invalid values indistinguishable from measurements.

TD-002 CRITICAL - No authoritative Framecount contract

Multiple programs sort or reinterpret rows. Gap/duplicate/wrap/invalid behavior is not centrally defined.

TD-003 HIGH - P001/P002 parser duplication and divergence

Common-origin code has diverged across multiple layers without shared tests/core.

TD-004 HIGH - P003 GUI/Core coupling

Cleaner behavior cannot be headlessly tested or orchestrated without refactor.

TD-005 HIGH - P004 Event Hint uses Row continuity

Large Framecount gaps can be merged into one Event Hint.

TD-006 HIGH - No versioned handoff schema

P002/P003/P004 compatibility depends on manual mapping and runtime behavior.

TD-007 MEDIUM/HIGH - Stage evaluation on fit rows

Stage improvement metrics can be optimistic. Cross-segment/log validation is not yet formalized.

TD-008 MEDIUM - P004 responsibility expansion

Static analysis, association, high-risk, event processing, and reporting are mixed.

TD-009 MEDIUM - Full-file memory processing

P003 reads full files; source data can exceed 1 GB. Performance validation is still required.


8. FORMAL DECISIONS v0.1

D001 KEEP - P002 physical decode math

Reason: real Synthetic tests confirmed central decode behavior. Refactor around it; do not rewrite blindly.

D002 MODIFY - P001/P002 shared parser architecture

Decision: future P001 and P002 must call one shared CAN/DBC Core. P001 remains optional discovery UI.

D003 REJECT AS FORMAL POLICY - Unlimited untracked Forward Fill

Current behavior may remain only for legacy compatibility until replacement. Future contract requires explicit limit/age/provenance.

D004 REJECT AS FORMAL POLICY - Generic Blank/NaN to Zero

Zero can be a valid measurement. Invalid values differ by Signal. Future missing/invalid policy must be explicit and per-signal capable.

D005 KEEP/MODIFY - Stage1/Stage2 mathematical implementation

Residual behavior is retained provisionally. Baseline output contract, sample gate, and validation split require clarification/experiments.

D006 REJECT - Row-only Event continuity

Future Event Candidate merging must use Framecount-aware rules.

D007 KEEP - P004 Event Hint as investigation aid

It is useful for reducing review scope, but is not confirmed Event detection or root cause.

D008 DEFER - SysID/AR/ARX expansion

Do not add dynamic models until canonical data, Framecount, target association, baseline, threshold, and Event output are stable.

D009 KEEP - Manual engineering verification

Video/CAN/tracking review remains required for final engineering hypotheses.


9. TARGET PROGRAM RESPONSIBILITIES

Shared CAN/DBC Core

- CSV header parsing primitives;
- DBC message/signal parsing;
- byte extraction;
- endian/signed/scale/offset decode;
- channel/ID/signal selection;
- tested error handling;
- no GUI.

P001 Catalog UI

- calls Shared CAN/DBC Core;
- displays available structure;
- no independent decode fork.

P002 Decoder UI/Core

- calls Shared CAN/DBC Core;
- outputs decoded updates plus provenance;
- does not silently own final missing-value/alignment policy.

Canonical Dataset Builder (P003 replacement core)

- validates schema;
- applies Framecount contract;
- sorts deterministically;
- preserves Source Row and update state;
- applies explicit per-signal invalid/imputation rules;
- outputs machine-readable manifest and validation report;
- callable without GUI.

P003 GUI Adapter

- collects configuration;
- calls Canonical Dataset Builder;
- contains no independent cleaning mathematics.

P004 Analysis Core

- consumes Canonical Dataset Contract;
- computes approved discrepancy/baseline/residual and investigation statistics;
- does not own generic data cleaning.

Event Candidate Processor

- threshold/persistence/Framecount-gap split/merge;
- start/peak/end Framecount;
- review-window export;
- no automatic root-cause claim.

Report Adapter

- Excel/CSV documentation and user-facing explanations;
- does not change analysis results.

Orchestrator (later)

- runs approved modules/configurations;
- writes run manifest;
- checks version compatibility;
- not implemented before callable cores and contracts exist.
