T004B - P002 CHARACTERIZATION TEST PROGRAM DESIGN

本文件是程序设计书，不是执行结果。


一、用户准备

1. 在公司Copilot Web Chat中新开一个对话。

2. 向Copilot Chat同时提供两个附件：

- matlab_can_time_order_decoder_gui.py
- 本设计书

3. 给Copilot以下指令：

把matlab_can_time_order_decoder_gui.py作为只读实现参考。
严格按照T004B程序设计书生成完整测试程序，并以可下载Python文件
t004b_p002_characterization_test.py返回。
必须调用真实P002实现，不得复制或重新实现Decoder算法。
不要运行程序，不要预测测试结果，不要直接返回Packet。
不要只在Chat中粘贴代码块；必须生成可下载的.py文件附件。


二、Copilot唯一交付物

Copilot只负责生成并返回可下载文件：

t004b_p002_characterization_test.py

禁止Copilot：

- 修改P002；
- 把修改后的P002作为交付物；
- 在Chat中直接返回Packet；
- 声称测试已经运行；
- 只返回代码文本而不生成Python文件；
- 使用公司真实CSV或DBC；
- 复制P002算法到测试脚本并测试复制品；
- 跳过Case后仍返回PASS；
- 创建自然语言报告。


三、测试程序目标

生成一个可由用户直接运行的Python程序：

python t004b_p002_characterization_test.py

程序运行时必须：

1. 找到同目录的P002；
2. 计算P002运行前SHA-256；
3. 使用tempfile创建完全Synthetic的CSV、CH1 DBC、CH2 DBC和输出目录；
4. 调用真实P002代码完成Decode；
5. 读取P002实际DataFrame或实际输出CSV；
6. 根据实际输出判断Q1-Q20；
7. 再次计算P002 SHA-256；
8. 确认P002未被修改；
9. 清理Synthetic临时文件；
10. Terminal只输出Block Packet。


四、实现边界

允许：

- importlib载入P002；
- 调用P002已有函数、类或GUI callback背后的Core；
- 在测试程序中创建最小Wrapper，把文件路径和GUI参数传给真实P002；
- 如果P002必须通过GUI对象运行，可以在不显示窗口的情况下构造必要对象；
- 使用pandas读取P002真实输出；
- 使用Python标准库生成Synthetic文件。

禁止：

- 在测试脚本中重新编写DBC Parser；
- 在测试脚本中重新实现bit decode、endian、signed、scale、offset；
- 在测试脚本中重新实现P002的wide expansion、sort或forward fill；
- 只根据源代码if语句填写Packet；
- 没有调用真实P002就把Q4标记为成功。

如果真实P002无法从测试程序调用：

- Q4填1；
- 相关未运行项填0；
- 不伪造输出。


五、Synthetic DBC要求

使用通用测试名称，不使用公司Signal。

CH1至少定义：

- CAN ID 0x100；
- 8 Byte Message；
- little-endian unsigned Signal；
- signed Signal；
- big-endian Signal；
- scale=2、offset=1的Signal；
- 名为SHARED_SIGNAL的Signal；
- 一个DBC min/max范围小于某个Synthetic输入值的Signal。

CH2至少定义：

- 同一个CAN ID 0x100；
- 与CH1不同的Message/Signal；
- 名为SHARED_SIGNAL的Signal，用于测试碰撞；
- CAN ID 0x200；
- Framecount Signal。

DBC语法和start bit必须适配P002当前Custom Parser。


六、Synthetic MATLAB CAN CSV要求

Header必须使用P002实际支持的格式，例如：

ID100_1_8_0
ID100_1_8_1
...
ID100_2_8_0
...
ID200_2_8_0
...

必须包含P002实际识别的DateTime和Time[us]列名。

Synthetic Rows至少覆盖：

1. Framecount=100更新；
2. CH1 0x100正常更新；
3. CH2 0x100正常更新；
4. 两行具有完全相同DateTime/Time[us]，但Signal值不同；
5. 一个Message只有部分Byte，其余Byte为空；
6. Framecount=101更新；
7. 普通Signal不更新但Framecount继续推进；
8. Framecount从101直接跳到110；
9. 一个物理值超出DBC min/max；
10. CH1与CH2的SHARED_SIGNAL使用不同数值。

八、Terminal输出格式

程序运行完成后只输出：

T004B1-5
A1-Q1Q2Q3Q4
A2-Q5Q6Q7Q8
A3-Q9Q10Q11Q12
A4-Q13Q14Q15Q16
A5-Q17Q18Q19Q20
C-x

规则：

- 使用连字符；
- 每个A Block恰好4个单字符Code；
- x为Q1-Q20总和modulo 10；
- 不在Terminal打印公司路径、Hash、源码或Synthetic数据；
- 调试详情仅写入本地T004B_TEST_LOG.txt；
- 即使部分Case失败，也尽量输出完整Packet；
- 无法确认使用0，不得猜测。
