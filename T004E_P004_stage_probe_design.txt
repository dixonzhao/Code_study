T004E - P004 STAGE CORE TARGETED PROBE DESIGN

本文件是测试程序设计书，不是执行结果。


0. 任务背景

System Under Test：

sre_cam_stage_residual_error_analyzer_v4_5_3_jp_compact_excel_notes.py

简称P004。它是现有Stage Residual Analyzer GUI，不是Decoder或Cleaner。

T004D真实运行结果已经确认：

- P004文件存在、可以Import；
- Reference-Target Discrepancy方向正确；
- Existing Discrepancy模式与直接计算一致；
- Pearson、Spearman、NMI能够得到有效结果；
- Lag按DataFrame Row计算，不按Framecount差计算；
- Candidate和High-Risk测试可以运行；
- 相邻Rows即使跨越大Framecount Gap，也会被合并为同一Event Hint；
- Stage1、Stage2在T004D中没有真正执行；
- P004真实Core只完成了Partial Execution；
- 报告、Event start/end/peak只得到Partial结果；
- P004源文件Hash未改变。

T004E只解决一个问题：

确定P004中Stage1/Stage2真实执行路径、数学行为和耦合边界。

不得在本任务中修改P004、优化算法、修复Bug或提出新算法。


1. 公司端提供给Copilot Web Chat的附件

必须同时提供：

1. P004真实源文件；
2. T004D生成的测试程序（如仍保留）；
3. T004D_TEST_LOG.txt（如已生成）；
4. 本T004E设计书。

如果2或3不存在，Copilot仍应只根据P004源文件和本设计书继续，不得中止。


2. 给Copilot的任务

请把P004源文件作为只读System Under Test。

先检查P004实际函数、类、GUI callback、参数和T004D失败位置，然后直接在Chat中输出完整Python源码：

t004e_p004_stage_probe.py

只输出一个完整Python代码块，不要输出解释、摘要、CODE_WRITTEN、文件链接或预测Packet。

Copilot只负责生成测试程序，不运行程序，不修改P004。


3. 测试程序强制要求

测试程序必须：

1. 在同一目录寻找精确P004文件名；
2. 计算P004运行前SHA-256；
3. Import真实P004；
4. 通过静态检查确定Stage逻辑位于纯函数、类方法、GUI callback还是多个位置；
5. 优先调用真实Stage Core；
6. 如果只能通过Analyzer对象或GUI callback到达Stage逻辑，可以创建最小对象、最小Tk变量替身或临时输入文件，但不得启动mainloop；
7. 可以Monkeypatch文件选择框、messagebox和输出目录，避免人工点击；
8. 必须调用真实P004 Stage数学实现；
9. 禁止在测试程序中重新实现Conditional Median、Stage Residual或Stage2算法；
10. 期望值必须使用本设计书给出的固定数值直接断言；
11. 无法运行的项目填0，不得猜测；
12. Debug写入本地T004E_TEST_LOG.txt；
13. Terminal最后只输出Machine Packet；
14. 再次计算P004 Hash，确认未修改。


4. 固定Synthetic Case

至少创建9行DataFrame，使用以下通用列：

- Framecount
- reference_distance
- target_distance
- stage1_axis
- stage2_axis

固定Raw Discrepancy定义：

reference_distance - target_distance

固定数据逻辑：

Stage1 Group A：4行

- stage1_axis位于同一个明确Bin；
- Raw Discrepancy为1, 1, 3, 3；
- 期望Stage1 Conditional Median为2；
- 期望Stage1 Residual为-1, -1, 1, 1。

Stage1 Group B：4行

- stage1_axis位于另一个明确Bin；
- Raw Discrepancy为11, 11, 13, 13；
- 期望Stage1 Conditional Median为12；
- 期望Stage1 Residual为-1, -1, 1, 1。

Sparse Group C：1行

- stage1_axis位于独立Bin；
- Raw Discrepancy为100；
- 用于观察P004是否直接计算单样本Median、拒绝该Bin或使用Fallback。

Stage2 Axis：

- 在Group A和B内部，前两行为Low Bin，后两行为High Bin；
- Stage1 Residual在Low Bin均为-1；
- Stage1 Residual在High Bin均为1；
- 期望Stage2 Baseline分别为-1和1；
- 期望8个主要样本的Final Residual全部为0。

可以根据P004真实参数API选择固定Bin Width或Bin配置，使上述分组成立，但不得改变Raw Discrepancy和期望值。


5. 必须执行的测试

CASE A - STAGE1 EXECUTION PATH

- 查明Stage1代码实际位于何处；
- 调用真实Stage1；
- 检查Baseline 2和12；
- 检查Residual -1,-1,1,1；
- 检查结果只存在输出文件、只存在内存，还是二者都有。

CASE B - STAGE2 EXECUTION PATH

- 在真实Stage1输出上调用真实Stage2；
- 检查Stage2 Baseline -1和1；
- 检查主要8行Final Residual为0；
- 检查结果可访问方式。

CASE C - SPARSE BIN AND EVALUATION CONTRACT

- 检查单样本Sparse Bin行为；
- 检查是否存在Minimum Sample Count Gate；
- 检查Baseline拟合和Stage Effect Metrics是否使用同一批Rows；
- 只检查现有实现，不评价算法好坏。

CASE D - T004D PARTIAL CAUSE

- 根据T004D程序、Log和P004 API，判断上一次Stage未执行的直接原因；
- 区分测试Wrapper/API不匹配、GUI耦合、P004真实异常、依赖/输出问题或多个原因。
