VRQ001 视频人工复核指南 v0.1


一、复核目的

本轮人工复核只回答一个核心问题：

    Recognition_ID选择出的Camera候选目标，是否与视频和Camera Tracking中
    可见的目标行为一致或至少在工程上合理？

本轮不负责：

- 判断Root Cause；
- 判断Camera或Radar究竟谁错；
- 将Radar当作Ground Truth；
- 根据Camera与Radar距离是否接近来选择Camera目标；
- 估计总体异常发生率或误报率。

Review Queue是有意富集的检查样本，其中包含：

- Camera ID切换；
- 稀有Camera ID；
- Recognition冲突；
- 数据质量例外；
- Provisional Discrepancy正负尾部；
- 正常对照区间。

因此，这些样本不能代表真实数据中的自然发生比例。


二、需要使用的文件

人工填写：

    video_review_queue_v0_1.csv

完成后使用以下程序压缩回传：

    video_review_label_encoder_v0_2.py

Encoder只读取填好的CSV，不会修改CSV。


三、逐行复核步骤

按照ReviewID顺序复核：

    Q001、Q002、Q003……

不要重新排序、删除或增加Review Queue中的行。

每一行执行以下步骤：

1. 打开Stereo Camera视频或StcamViewer。
2. 跳转到该行的WindowStartFrame。
3. 从WindowStartFrame开始观察，经过KeyFrame，一直看到WindowEndFrame。
4. 重点观察KeyFrame前后目标是否连续、是否发生真实切换、Camera Tracking是否稳定。
5. 可以结合以下信息判断：
   - 视频中的目标；
   - Camera Overlay或Tracking框；
   - SelectedCameraCANID；
   - Camera目标连续性；
   - LateralPosition；
   - SwitchingFlag；
   - Camera波形或Target状态。
6. 不得因为Selected Camera距离与Radar距离接近，就判断Camera选择正确。
7. 在CSV中填写下面五个必填数字标签。
8. 只有确实有帮助时，才在ReviewerNotes中写简短备注。

本轮的RadarMeasurementPlausible可以留空。Radar没有与Camera对应的Object ID，
并且Radar目前只是暂定工程Reference。


四、五个必填数字标签

每个字段只能填写一个数字：0、1、2或3。


4.1 CameraTargetVisible

判断视频中的目标是否能够被用于本轮复核。

0 = NOT_REVIEWED
    尚未复核。

1 = CLEARLY_VISIBLE
    目标清楚可见，可以判断目标连续性和切换行为。

2 = PARTIALLY_VISIBLE
    目标部分可见，仍能提供一些判断依据。

3 = NOT_VISIBLE_OR_UNREVIEWABLE
    目标不可见，或者当前视频/Overlay无法支持复核。


4.2 SelectedCameraMatchesDisplayedTarget

判断Recognition_ID所选Camera候选，是否与视频或Camera Tracking中显示的目标
在工程上相符。

这只是Selector Plausibility判断，不是Camera与Radar物理目标一致性的证明。

0 = NOT_REVIEWED
    尚未复核。

1 = PLAUSIBLE_MATCH
    选择合理，所选Camera候选与可见目标或Tracking行为相符。

2 = IMPLAUSIBLE_MATCH
    有明确证据表明所选Camera候选与可见目标或Tracking行为不符。

3 = UNCERTAIN
    现有视频、Overlay或信号不足以判断。


4.3 VisibleTargetSwitchOrJump

判断KeyFrame附近是否发生可见的目标切换或不合理跳变。

0 = NOT_APPLICABLE_OR_NOT_REVIEWED
    不适用、尚未复核，或该窗口无法判断。

1 = NO_VISIBLE_SWITCH_OR_JUMP
    没有观察到明显目标切换或跳变。

2 = VISIBLE_PLAUSIBLE_TARGET_SWITCH
    观察到目标切换，但场景能够合理解释，例如：
    - Cut-in；
    - Cut-out；
    - 超车；
    - 遮挡；
    - 前车变化；
    - Reference Target正常切换。

3 = VISIBLE_SUSPICIOUS_JUMP
    Camera数值或目标选择发生明显跳变，但视频中没有足以解释它的场景变化，
    或跳变行为看起来不连续、不合理。


4.4 CameraTrackingStable

判断Camera Tracking在窗口内的稳定程度。

0 = NOT_REVIEWED_OR_UNCLEAR
    尚未复核或无法判断。

1 = STABLE
    目标Tracking连续、稳定。

2 = UNSTABLE
    Tracking存在抖动、明显不连续或状态不稳定。

3 = LOST
    目标Tracking丢失。


4.5 VideoReviewResult

这是该Review Case对Camera Selector的总体判断。

0 = NOT_REVIEWED
    尚未完成该行复核。

1 = SELECTOR_SUPPORTED
    Recognition_ID所选Camera候选与视频、Camera Tracking和目标连续性相符，
    当前证据支持Selector。

2 = SELECTOR_CONTRADICTED
    有正面证据表明Recognition_ID所选Camera候选与视频或Camera Tracking不符。

3 = INSUFFICIENT_EVIDENCE
    因为视频、Overlay、遮挡、场景复杂或信号不足，无法支持或否定Selector。

注意：

Camera与Radar差值很大，本身不足以填写2。

只有当视频或Camera Tracking提供明确反证时，才填写：

    VideoReviewResult = 2


五、无法复核时怎么填

如果某个Case完全看不清或无法使用当前视频判断，不要留空，填写：

    CameraTargetVisible = 3
    SelectedCameraMatchesDisplayedTarget = 3
    VisibleTargetSwitchOrJump = 0
    CameraTrackingStable = 0
    VideoReviewResult = 3


六、不同ReasonCodes应该看什么


6.1 VALID_START_ANCHOR

- 确认FrameCount约6500的视频位置可用；
- 确认从这一带开始进入正式行驶和分析区间；
- 检查视频与FrameCount是否明显错位。


6.2 INVESTIGATION_ANCHOR

- 重点观察FrameCount约38239前后；
- 检查是否出现此前人工观察到的Camera 0x221/0x251跳变；
- 38239只是人工调查锚点，不是Selector硬编码边界。


6.3 CAMERA_ID_COVERAGE

- 检查该SelectedCameraCANID是否对应一个合理的Camera目标；
- 稀有ID即使只出现很短时间，也不能自动判断为错误；
- 不得因为某个ID罕见，就把它替换成0x221或0x251。


6.4 RECOGNITION_TRANSITION

- 观察KeyFrame前后的目标身份；
- 检查是否发生真实、合理的Reference Target变化；
- 区分合理Target Switch和没有场景依据的数值跳变。


6.5 RECOGNITION_CONFLICT

- 检查同一FrameCount内Recognition状态冲突时，视频中是否出现目标不连续；
- 如果视频无法分辨Frame内部的先后顺序，不要强行下结论；
- 此时优先使用UNCERTAIN或INSUFFICIENT_EVIDENCE。


6.6 MULTI_CAMERA_UPDATE

- 同一FrameCount内多个Camera候选更新不一定是错误；
- 只判断Recognition_ID最终选择出的候选是否合理；
- 不要因为“多个候选同时更新”就自动判定异常。


6.7 MAX_SELECTED_SIGNAL_AGE

- 检查SignalAgeFrames = 1时目标连续性是否仍然正常；
- 这用于验证当前正式采用的Maximum SignalAgeFrames = 1。


6.8 SELECTION_STATUS_EXCEPTION

如果是NO_CANDIDATE_UPDATE_OBSERVED：

- 检查Selected Camera候选是否尚未建立可用Track或状态；
- 不要自动用其他Camera候选替代。

如果是INVALID_SELECTED_DISTANCE：

- 记录视频中能够观察到的情况；
- 不要为了得到Distance而替换Camera候选。


6.9 POSITIVE_DISCREPANCY_EXTREME和NEGATIVE_DISCREPANCY_EXTREME

- 检查Camera Tracking、遮挡、弯道、多目标和目标关联变化；
- 不要因为属于Discrepancy Tail就直接叫作异常；
- 当前Provisional Discrepancy仍包含distance-dependent baseline。


6.10 STABLE_CONTROL和INVESTIGATION_CONTROL

- 对正常对照样本使用与可疑样本相同的检查标准；
- 正常对照用于避免只查看高差值和切换窗口造成判断偏差。


七、ReviewerNotes怎么写

ReviewerNotes不是必填项。

建议只在以下情况写几个英文单词或短句：

- cut-in
- cut-out
- multi target
- curve
- occlusion
- Camera overlay unavailable
- target identity unclear
- visible target stable despite large discrepancy
- visible jump without scene change

不要在备注里写没有证据支持的Root Cause结论。


八、如何压缩并回传给ChatGPT/Codex

全部ReviewID填写完成后：

1. 保存CSV。
2. 如果Excel询问是否保持CSV格式，选择保持CSV格式。
3. 关闭Excel，避免文件仍被占用或保存未完成。
4. 运行video_review_label_encoder_v0_2.py。
5. 选择填好的video_review_queue_v0_1.csv。
6. Encoder会检查：
   - ReviewID是否仍为Q001、Q002……的原始顺序；
   - 是否存在删除、重复或重排行；
   - 五个必填字段是否都只包含0、1、2、3；
   - 输入CSV在编码过程中是否保持不变。
7. 将终端输出的VRQ001L2 Packet完整输入给ChatGPT/Codex。

输出示例：

    VRQ001L2-6-N24
    B1-166-155-155-155
    B2-155-166-155-259
    B3-155-165-156-355
    B4-155-155-165-156
    B5-165-155-155-155
    B6-166-155-155-155
    K1-01C97-01CA1-01CC0-01D22
    K2-01D80-01E10-01E90-01F04
    R1-014-400-020-100
    R2-004-040-080-200
    S1-2222
    S2-2222
    S3-2211
    C-F
    X2-Q008
    X3-Q012

其中B记录每个Q的五个标签，K记录KeyFrame，R记录ReasonCodes。用户不需要
理解或手动解码这些字符。

其中：

- X2列出VideoReviewResult = 2，即Selector被证据否定的ReviewID；
- X3列出VideoReviewResult = 3，即证据不足的ReviewID。

只需要为X2和X3中的少数Case额外输入短备注，例如：

    N-Q008-visible jump without scene change
    N-Q012-Camera overlay unavailable

不要上传或传输填好的公司CSV。


九、VRQ001完成标准

VRQ001人工复核完成需要满足：

- 每个ReviewID都填写五个必填数字标签；
- Selector被否定或证据不足的Case在必要时有短备注；
- Encoder没有报告空白、非法值、删除行、重复行或排序变化；
- VRQ001L2 Packet已经返回；
- 能够根据复核结果对Camera Selector和Maximum SignalAgeFrames = 1作出：
  Keep、Modify或Reject决定。
