T010 - CANONICAL DATASET CONTRACT v0.2

Status: FORMAL PROJECT CONTRACT FOR R1
Date: 2026-07-21
Applies to: P002 refactor, P003 replacement Core, P004 Core, Event Processor, later SysID


1. PURPOSE

This contract prevents decoded CAN updates, missing values, invalid values, carried values, Framecount alignment, and analysis rows from being mixed without traceability.

It replaces the current implicit behavior:

P002 Wide CSV
-> Unlimited Forward Fill
-> P003 generic Blank/NaN to Zero
-> P004 treats rows as ordinary measurements

with two explicit layers:

Layer A - Decoded Update Table

Layer B - Canonical Frame Table


2. EVIDENCE SCOPE

BC010 Profiler input:

- Raw MATLAB CAN CSV: complete file;
- P002 Decoded CSV: partial/subset file, not full-video output.

Therefore:

- Raw row-count class is representative for the selected Raw file;
- Decoded percentage/rate values must not be generalized to the full video;
- Decoded structural observations are accepted when they demonstrate that a behavior exists;
- absence in the partial Decoded file does not prove absence in full data.

BC010 confirmed observations:

- complete Raw file streamed successfully;
- Raw file contains at least 100,000 and fewer than 500,000 rows in the profiled selection;
- Profiler did not recognize the Raw Framecount CAN header, so Raw Framecount update density remains UNKNOWN;
- partial P002 Decoded file loaded successfully;
- Framecount column was found through a named/fuzzy resolution path;
- Decoded Framecount values were at least 99.9 percent non-missing in the partial file;
- Framecount values were integer-like;
- duplicate Framecount rows were common/high in the partial file;
- maximum rows per Framecount was between 6 and 20;
- current Decoded file order was not monotonically increasing by Framecount;
- both Framecount gaps and backward/reset-like transitions were observed;
- observed positive gaps were no larger than 5 in this partial file;
- many selected Signals changed value within the same Framecount;
- Framecount zero was not observed in the partial file;
- many non-Framecount Signal columns contained zero;
- many columns contained repeated boundary/extreme candidate values;
- validity/status-like columns existed, but sensor ownership could not be reliably inferred from names;
- DateTime and Time[us] columns existed;
- source_row_id did not exist.

Interpretation limit:

- repeated zero or boundary values are investigation candidates, not automatically invalid values;
- backward Framecount transitions may result from current sorting, segment boundaries, resets, or mixed input; cause remains unresolved;
- no automatic Root Cause conclusion follows from this profile.


3. FORMAL ARCHITECTURE DECISIONS

D010 TWO-LAYER DATA MODEL - APPROVED

Layer A preserves asynchronous decoded updates.

Layer B provides one validated row per Framecount/segment for analysis and video linkage.

Reason:

- CAN updates are asynchronous;
- video review uses Framecount;
- a CAN update row is not an equally sampled time step;
- row adjacency is not Framecount continuity;
- later SysID requires an explicit sampling grid.

D011 P003 TRANSITION - APPROVED

Keep the current P003 GUI as a read-only Legacy Tool during transition.

Future P003 UI may remain, but its internal cleaning logic must be replaced by a callable Canonical Dataset Builder Core.

D012 FIRST IMPLEMENTATION PRIORITY - APPROVED

Implement a read-only Canonical Dataset Validator before transforming data or refactoring Decoder mathematics.

D013 SAME-FRAME COLLAPSE - NOT YET APPROVED

Because many Signals can change within one Framecount, do not select first/last/mean/median globally.

Preserve all Layer A updates. Layer B selection must be per-Signal and must expose update count/conflict/source identity.

D014 LEGACY SOURCE IDENTITY - APPROVED

For existing P002 output without original Raw source identity, create legacy_row_id only.

Do not label it source_row_id.

D015 FRAMECOUNT ORDER - APPROVED

Current formal analysis order is frame_segment_id + Framecount.

DateTime and Time[us] remain provenance only for this project stage.


4. NON-NEGOTIABLE DATA RULES

R1 Framecount is the current authoritative video/alignment axis.

R2 Missing, No Update, Invalid, Unavailable, Carried, Expired, and True Zero are distinct states.

R3 No generic Blank/NaN-to-Zero policy is allowed in the new Core.

R4 No Unlimited Forward Fill is allowed in the new Core.

R5 Carry/Forward Fill requires an explicit per-Signal rule and maximum age in Framecount units.

R6 Every Layer B value must trace to an original Layer A update or an explicit aggregation/carry rule.

R7 Invalid/missing Framecount must never be replaced with zero.

R8 Event continuity must use Framecount/segment rules, not DataFrame row adjacency.

R9 P004 consumes validated Layer B data and does not own generic cleaning.

R10 Legacy outputs must be tagged as legacy and cannot silently satisfy the canonical contract.


5. LAYER A - DECODED UPDATE TABLE

Grain:

One row represents one decoded Signal update from one Raw CAN source message.

If one message contains multiple Signals, each Signal update may be represented as a separate Layer A row sharing source_message_id.

Required fields:

- schema_version = canonical_update_v0.2;
- run_id;
- source_file_id without unapproved raw path;
- source_row_id;
- source_message_id;
- channel;
- can_id;
- dlc;
- signal_name;
- collision-safe signal_key = channel + can_id + signal_name;
- decoded_value;
- unit;
- decode_valid;
- invalid_reason;
- is_original_update = true;
- framecount_context, nullable;
- frame_segment_id, nullable until alignment;
- datetime_raw, optional provenance;
- time_us_raw, optional provenance.

Layer A prohibited behavior:

- no generic Forward Fill;
- no generic Zero Fill;
- no silent drop of invalid updates;
- no loss of source_row_id;
- no silent Signal collision;
- no use as equally sampled SysID input.


6. LAYER B - CANONICAL FRAME TABLE

Grain:

One row per valid Framecount within one frame_segment_id.

Primary key:

frame_segment_id + Framecount

Required identity fields:

- schema_version = canonical_frame_v0.2;
- run_id;
- frame_segment_id;
- Framecount integer;
- frame_sequence_index;
- frame_gap_from_previous;
- frame_contiguous;
- frame_source_count;
- frame_quality_status;
- frame_quality_reason.

For each selected Signal S:

- S: selected physical value, nullable;
- S__updated: original update exists in this frame;
- S__carried: value came from an earlier frame;
- S__age_frames: 0 for update, positive for carry, nullable if unavailable;
- S__valid;
- S__invalid_reason;
- S__source_row_id;
- S__update_count_in_frame;
- S__conflict.


7. FRAMECOUNT CONTRACT

Valid Framecount:

- integer-like;
- not created by missing-value fill;
- belongs to one explicit frame_segment_id.

Rows without valid Framecount:

- remain in Layer A with alignment status;
- do not enter valid Layer B;
- are counted in validation output.

Duplicate Framecount:

- expected at Layer A because multiple updates may occur in one video frame;
- prohibited as a duplicate primary key in Layer B;
- cannot be resolved by generic drop_duplicates.

Gap:

- every Layer B row records frame_gap_from_previous;
- Event Processor must split or flag windows according to an approved gap threshold.

Backward transition/reset/wrap:

- do not silently reorder it away before reporting;
- report transition position and magnitude;
- require segment decision before canonical build;
- create a new segment only by an explicit rule.


8. VALUE STATE CONTRACT

Minimum states:

- ORIGINAL_VALID_UPDATE;
- ORIGINAL_INVALID_UPDATE;
- NO_UPDATE_THIS_FRAME;
- CARRIED_VALID_WITHIN_AGE;
- CARRIED_EXPIRED;
- UNAVAILABLE;
- TRUE_ZERO.

Zero:

- treated as a numeric value by default;
- invalid only through a Signal-specific rule.

Sentinel:

- configured per signal_key or approved Signal group;
- repeated boundary values are candidates requiring validation, not automatic sentinels.

Carry:

- default carry_allowed = false;
- enabling carry requires physical reason, maximum age, expiry behavior, invalid-update behavior, and suitability for lag/SysID.


9. SAME-FRAME MULTIPLE UPDATE CONTRACT

Current default:

- preserve every update in Layer A;
- Layer B records update_count and conflict;
- no universal first/last/mean/median rule;
- formal analysis requires an explicit per-Signal rule.

Supported future rule types:

- LAST_VALID_BY_SOURCE_ORDER;
- FIRST_VALID_BY_SOURCE_ORDER;
- MEDIAN_WITHIN_FRAME;
- MEAN_WITHIN_FRAME;
- REJECT_IF_MULTIPLE;
- SIGNAL_SPECIFIC_CUSTOM.

The rule and its parameters must be recorded in the manifest.


10. SORT CONTRACT

Layer A identity order:

- source_file_id;
- source_row_id.

Sorting a view must not overwrite source identity.

Layer B formal order:

- frame_segment_id;
- Framecount.

Current DateTime/Time[us] columns cannot replace this order.


11. DATA QUALITY GATE

Before P004 accepts Layer B, validate:

- supported schema_version;
- required identity fields;
- Framecount integer-like;
- unique primary key;
- order monotonic within each segment;
- gap/reset summary;
- Reference/Target numeric compatibility;
- Unit known or warning recorded;
- invalid/missing/carried rate per selected Signal;
- maximum Signal age;
- no unapproved generic Zero Fill;
- no unapproved Unlimited Carry;
- target/reference roles and discrepancy direction recorded.

Gate outcomes:

PASS

PASS_WITH_WARNING

FAIL

On FAIL, P004 must not generate a normal-looking final report.

13. P004 INPUT CONTRACT

P004 Core receives:

- validated Layer B table;
- manifest/schema object;
- Framecount role;
- Reference role;
- Target role;
- discrepancy direction;
- Unit confirmation;

16. FIRST CONTROLLED IMPLEMENTATION

Name:

Canonical Dataset Validator v0.1

Type:

read-only standalone Python Core + minimal file-selection wrapper.

Input:

legacy P002 Decoded CSV.

It must NOT transform or overwrite input data.

It must inspect:

- Framecount detection/type/missingness;
- row-order monotonicity;
- duplicate rows per Framecount;
- Gap and backward/reset candidates;
- Source Row availability;
- Zero/NaN distribution;
- repeated boundary candidates;
- validity/status column candidates;
- schema/metadata gaps;
- estimated risk of using the file directly in P004.

Output:

- canonical_validation_manifest_v0_1.json;
- canonical_validation_summary_v0_1.csv;
- canonical_validation_log_v0_1.txt;
- terminal status packet.

It must classify:

- PASS only when canonical requirements are actually satisfied;
- PASS_WITH_WARNING for analyzable legacy input with explicit limitations;
- FAIL when Framecount/required roles are unusable.

The Validator does not certify target association, ground truth, or root cause.


17. DEFERRED FACTS

The following remain open but do not block the read-only Validator:

- exact Raw 0x620 Header spelling/update density;
- cause of observed backward Framecount transitions;
- Framecount wrap modulus/reset semantics;
- per-Signal validity/status ownership;
- per-Signal sentinel rules;
- per-Signal same-frame aggregation;
- per-Signal carry permission and age;
- full-video frequency/rate statistics.

These facts block Canonical Builder transformation, not Validator implementation.
