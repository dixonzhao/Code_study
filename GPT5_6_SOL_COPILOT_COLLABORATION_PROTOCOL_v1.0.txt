Camera–Radar Project
GPT-5.6 SOL Work / 用户 / 公司 Copilot 协作协议
Version: 1.0
Date: 2026-07-21

======================================================================
0. 文档目的
======================================================================

本文档定义 Camera–Radar Discrepancy Analysis and Anomaly
Localization Project 中三个角色的职责、任务传递方式、冲突处理规则和最终输出格式。

三个角色：

1. GPT-5.6 SOL Work：项目技术 Pilot、技术路线制定者、项目状态管理者。
2. 用户：公司环境中的工程执行者、实验负责人和合规边界确认者。
3. 公司 Copilot：代码实现工具，负责按已批准设计生成、调试和重构公司环境代码。

本项目必须保持单一技术决策源，避免不同 AI 对话分别修改数学定义和 Pipeline。


======================================================================
1. GPT-5.6 SOL Work 负责什么
======================================================================

GPT-5.6 SOL Work 负责“决定做什么、为什么做、如何验证”。

1.1 项目与架构职责

- 维护项目 Mission、Scope 和当前优先级。
- 维护唯一正式 Pipeline。
- 定义 Program Responsibility、输入输出接口和执行顺序。
- 区分 Bug、Design Gap 和 Known Limitation。
- 防止新功能使 Analyzer 持续膨胀或职责混乱。

1.2 算法职责

- 定义 raw discrepancy、baseline、residual、anomaly score、threshold 和 event。
- 设计数学模型、输入输出、假设条件、使用范围和局限。
- 判断是否需要 Stage、Block/Scenario、Rule-based Event、AR/ARX 或其他方法。
- 在现有方法足够时，拒绝不必要的复杂度。
- 不允许未经验证的方法直接进入正式 Pipeline。

1.3 实验与决策职责

- 给每个正式实验分配 E 编号。
- 定义实验目的、唯一改变项、评价指标和停止条件。
- 根据公司环境真实运行结果做 Keep / Reject / Modify 决策。
- 维护 Known Facts、Observations、Hypotheses、Open Questions、Experiment Log 和 Decision Log。
- 将 Next Actions 限制为当前最高优先级的 1—3 项。

1.4 GPT-5.6 SOL Work 不负责

- 不接收或保存公司 CAN、DBC、Video、代码或内部文档。
- 不在没有真实数据证据时伪造实验结果。
- 不替代用户做公司合规、安全和最终交付决定。
- 不直接编辑公司电脑中的真实代码或数据。


======================================================================
2. 用户负责什么
======================================================================

用户负责“在真实工程环境执行、观察、复核和反馈”。

2.1 工程执行

- 在公司批准的电脑、VS Code、Python、MATLAB 等环境中运行程序。
- 使用公司 Copilot 按 GPT-5.6 SOL Work 提供的 Task Packet 实现代码。
- 运行实验、生成报告、检查曲线、Video、CAN 和 Tracking 状态。
- 确认公司环境的真实列名、数据类型、单位、频率和业务定义。

2.2 证据反馈

- 手动转录允许离开公司环境的必要信息。
- 明确区分“真实观察”和“个人推测”。
- 按规定格式反馈数值结果、曲线形态、报错、运行时间和人工复核结论。
- 对公司业务定义和上长要求做最终确认。

2.3 合规职责

- 不将公司测试文件、视频、代码、DBC、截图或内部文档上传到个人 Mac、GitHub 或 GPT。
- 只在公司允许的范围内手动转录结构化信息。
- 确认从个人 GitHub 下载的文件不包含未批准内容。


======================================================================
3. 公司 Copilot 负责什么
======================================================================

公司 Copilot 负责“如何在公司环境把已批准设计实现成可运行代码”。

3.1 允许职责

- 按 Task Packet 生成 Python、MATLAB、C/C++ 或其他实现代码。
- 根据真实 DataFrame 和列名适配实现。
- 开发 GUI、输入检查、Excel/CSV 输出和绘图功能。
- 修复 Bug、异常处理、类型问题和性能问题。
- 在不改变数学意义和程序接口的前提下 Refactor。
- 生成测试，运行指定检查，报告实现结果。

3.2 禁止自行决定

未获得新 Task Packet 或 Decision 批准时，Copilot 不得自行：

- 新增统计方法或机器学习方法。
- 修改 raw discrepancy、baseline、residual、threshold 或 event 定义。
- 修改正式 Pipeline。
- 新增评价指标或改变指标的计算样本集。
- 把 correlation、NMI、lag 或 high-risk score 解释为因果证据。
- 把 Event Hint 解释为 Confirmed Event。
- 把 Radar 解释为 Ground Truth。
- 创建新 Analyzer 来临时规避当前接口或职责问题。

3.3 Copilot 可以提出异议

如果 Copilot 认为 Task Packet 存在技术风险，应：

1. 暂停对有争议部分的实现。
2. 指出具体的数据、数学、接口或代码冲突。
3. 说明不修改时将产生的可复现后果。
4. 提供最小修改建议，但不直接将建议合并到正式路线。
5. 等待用户将问题手动反馈给 GPT-5.6 SOL Work。

4. 两边如何传递任务、代码和分析结果
======================================================================

4.1 信息通道

方向 A：个人 Mac / GPT → 公司电脑

1. GPT-5.6 SOL Work 生成通用 TXT、Markdown、伪代码或无公司信息模板。
2. 用户将文件上传到个人 GitHub。
3. 公司电脑匿名下载文件。
4. 用户在公司环境确认内容后提交给 Copilot。

方向 B：公司电脑 → 个人 Mac / GPT

1. 不使用电子文件传输。
2. 由用户手动转录必要的非敏感信息。
3. 只转录判断路线必需的结构、数值、形态、报错和人工复核结论。

GitHub 是单向知识输入通道，不是公司文件同步盘。

4.2 GPT 发给 Copilot 的标准 Task Packet

每个任务使用 T 编号，格式如下：

---------------------------------------------------------------------
TASK_ID: T001
TITLE:
STATUS: Approved for implementation

GOAL:
本任务要解决的唯一问题。

CONTEXT:
与当前程序和 Pipeline 相关的最小背景。

INPUT:
输入文件、DataFrame、必需列、单位和 dtype。

MATHEMATICAL_DEFINITION:
已批准数学定义。如无数学变化，写“不修改现有数学定义”。

IMPLEMENTATION_SCOPE:
Copilot 允许新增或修改的函数、GUI、报告或文件。

CONSTRAINTS:
不得改动的接口、列、数学定义和兼容性要求。

NON_GOALS:
本任务明确不处理的内容。

OUTPUT:
应生成的代码、报告、日志或结果字段。

VERIFICATION:
必须运行的测试和检查。

DONE_WHEN:
可验证的完成条件。

REPORT_BACK:
Copilot 完成后必须返回的摘要。
---------------------------------------------------------------------

4.3 Copilot 返回实现结果的标准格式

---------------------------------------------------------------------
TASK_ID:
IMPLEMENTATION_STATUS: Complete / Partial / Blocked

FILES_CHANGED:
文件名和版本。

FUNCTIONS_CHANGED:
新增或修改的函数名。

BEHAVIOR_CHANGED:
用一至三句话说明实际行为变化。

INTERFACE_CHANGED:
No / Yes。如 Yes，必须说明原因，不得自动批准。

MATHEMATICS_CHANGED:
必须为 No。如无法保持 No，任务应转为 Blocked。

TESTS_RUN:
测试名称、输入类型、通过/失败。

KNOWN_LIMITATIONS:
仍然存在的局限。

USER_ACTION:
用户需要在公司电脑上运行和检查的内容。
---------------------------------------------------------------------

4.4 用户反馈实验/分析结果的标准格式

---------------------------------------------------------------------
EXPERIMENT_ID:
TASK_ID:
DATA_SCOPE:
例如：前半段/全视频，约多少 datapoints。不输入内部文件名。

PARAMETERS:
本次实验的关键参数。

NUMERIC_RESULTS:
Q95、candidate rate、event count、runtime 等预先指定指标。

OBSERVATIONS:
曲线、Video、Tracking 或报告中真实观察到的现象。

ERRORS:
报错原文或简要转录。

MANUAL_REVIEW:
人工视频复核结论。

USER_HYPOTHESIS:
可选。必须标记为推测，不与 Observation 混合。
---------------------------------------------------------------------

5. 出现结论冲突时如何交叉验证
======================================================================

5.1 基本原则

- 没有任何 AI 的权威高于公司环境可复现的真实证据。
- GPT-5.6 SOL Work 是正式技术路线维护者，但其假设也必须经实验检验。
- Copilot 可以识别实现冲突，但不得自行替换数学定义。
- 用户对真实环境、业务要求和合规边界拥有最终确认权。

5.2 冲突分类

A. 实现冲突
代码无法按 Task Packet 运行，或真实 DataFrame 与规定接口不一致。

处理：Copilot 报告精确冲突，用户反馈真实结构，GPT 修改接口设计或发布新 Task Packet。

B. 数学或算法冲突
Copilot 认为应更换公式、统计方法或阈值。

处理：不直接改代码。将建议记录为 Hypothesis/Alternative，由 GPT 设计对照实验，根据结果进入 Decision Log。

C. 模型结论与 Video/CAN 观察冲突
例如模型输出高 residual，但 Camera tracking 人工观察仍稳定。

处理：保留观察，不把模型分数解释为 root cause。分别检查目标关联、时间同步、距离定义、reference 不确定性和 baseline 不完整性。

D. 业务或合规冲突
AI 设计与公司规定、上长要求或数据安全要求冲突。

处理：立即停止相关实现，以用户和公司确认为准。

5.3 标准交叉验证流程

1. 冻结争议点，暂不修改正式 Pipeline。
2. 用一句话写明两个相互冲突的结论。
3. 分离 Known Fact、Observation、Model Result 和 Hypothesis。
4. 设计只改变一个主要项的最小对照实验。
5. 预先规定评价指标和通过/失败标准。
6. Copilot 只实现实验所需的最小代码改动。
7. 用户在公司环境运行并反馈结果。
8. GPT 根据证据作出 Keep / Reject / Modify 决策。
9. 将决定记入 Decision Log，然后才更新 Pipeline 和代码任务。

6. 最终成果采用什么格式输出
======================================================================

6.1 项目管理成果（个人 Mac / Project Control）

- `00_PROJECT_STATE.md`
- `01_PROGRAM_INVENTORY.md`
- `02_INTERFACE_REGISTRY.md`
- `03_METHOD_STATUS.md`
- `04_LOGS.md`
- `05_OPEN_QUESTIONS_AND_NEXT_ACTIONS.md`
- Copilot 任务包：`Txxx_<short_title>.txt`

这些文件不包含公司数据、代码或内部信息。

6.2 公司代码成果（只保留在公司环境）

- Python / MATLAB / C/C++ 源代码
- GUI 程序
- 配置文件
- 测试脚本
- 公司环境中的版本记录

代码不上传到个人 GitHub 或 GPT。

6.3 第一阶段工程交付

主交付文件：

1. `anomaly_event_list.csv`

建议字段：

- event_id
- start_framecount
- peak_framecount
- end_framecount
- start_video_time
- peak_video_time
- end_video_time
- max_abs_residual
- anomaly_score
- review_priority
- reference_status
- manual_review_status
- manual_review_label
- investigation_notes

2. `camera_radar_anomaly_review.xlsx`

建议 Sheet：

- `00_summary`
- `01_event_list`
- `02_event_detail`
- `03_stage_diagnostics`
- `04_data_quality`
- `05_method_notes`

3. `project_state_summary.txt` 或 `.md`

内容：

- 任务和数据范围
- 当前 Pipeline
- 方法与假设
- 已知限制
- 核心观察
- 尚未解决问题
- 下一步建议

6.4 术语和证据要求

最终输出必须严格区分：

- Camera–Radar discrepancy
- Baseline
- Residual
- Anomaly score
- Event Hint / Review Window
- Confirmed Event
- Model Result
- Observation
- Hypothesis
- Root Cause

在没有 ground truth 时，不得将 Camera–Radar discrepancy 称为 Camera 真实误差。

7. 标准工作流
======================================================================

问题/缺口输入
→ GPT-5.6 SOL Work 进行问题定义
→ GPT-5.6 SOL Work 设计方案或实验
→ Decision/Task Packet 发布
→ 用户将通用文件导入公司环境
→ Copilot 实现代码
→ 用户运行与观察
→ 用户手动反馈标准结果摘要
→ GPT-5.6 SOL Work 分析证据
→ Experiment Log / Decision Log
→ 更新正式 Pipeline
→ 发布下一个 Task Packet


======================================================================
8. 当前特别指令
======================================================================

- 当前阶段先完成 Program Inventory、Program Responsibility、Interface Registry 和端到端执行顺序确认。
- 盘点完成前，不创建新 Analyzer。
- 当前不开发 LDW、TSR、Resume、HMI 或 SysID 新模块。
- 当前不修改现有数学定义。
- Copilot 应先帮助用户审查现有程序结构和输入输出，而不是推荐新算法。


======================================================================
9. 决策优先级
======================================================================

1. 公司合规、安全和上长已确认的业务要求。
2. 公司环境中可复现的真实数据与实验证据。
3. Project Decision Log 中已批准的技术路线和数学定义。
4. GPT-5.6 SOL Work 发布的当前 Task Packet。
5. Copilot 的实现建议。

如 Copilot 建议与已批准路线不一致，Copilot 应报告冲突，不得静默修改正式定义。
