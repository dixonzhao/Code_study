======================================================================
2. STEP-BY-STEP INSPECTION PROCEDURE
======================================================================

STEP 1 — Confirm the inputs

Record:

- exact filename;
- file size in bytes if visible;
- total line count;
- text encoding if visible;
- top-level imports;
- class names;
- function names;
- whether an if __name__ == "__main__" guard exists;
- whether importing the file appears capable of starting GUI or file I/O.

Do not execute the programs to discover this.

STEP 2 — Build a source inventory

For every class/function/method, record:

- identifier;
- line range or source anchor;
- current responsibility;
- input;
- output;
- global-state dependency;
- GUI-widget dependency;
- file-system side effect;
- reusable as pure logic: YES / PARTIAL / NO / UNKNOWN.

Very small callback/wrapper functions may be grouped only when they have the
same responsibility and the group remains traceable.

STEP 3 — Trace each legacy runtime flow

Describe the code-visible flow separately:

P001:
source/header detection
→ DBC loading/parsing
→ CAN ID/channel/signal inventory
→ GUI presentation
→ catalog output

P002:
source/header detection
→ DBC loading/parsing
→ selected-message byte collection
→ bit decoding and physical conversion
→ wide expansion
→ ordering
→ missing/update handling
→ output

P003:
decoded-file loading
→ column selection
→ type conversion
→ invalid/missing handling
→ ordering
→ output

If actual flow differs, report the source-visible flow and mark the expected
step as ABSENT.

STEP 4 — Create a duplication matrix

Compare P001, P002, and P003 for:

13. scale/offset conversion;
14. column naming;
15. sorting;
16. missing-value processing;
17. output naming;
18. file writing;
19. progress/error reporting.

For every duplicated item, identify:

- implementations involved;
- whether behavior is equivalent;
- which implementation has clearer separation from GUI;
- which implementation should be the extraction reference;
- unresolved risk.

Do not select an owner merely because one file is newer or larger.

STEP 5 — Audit Data Contract conflicts

Check the source against every rule below.

Order and provenance:

- preserve SourceRowIndex;
- do not claim DateTime/Time[us] is trustworthy;
- use FrameCount as the current video synchronization key;
- distinguish source update order from FrameCount order;
- record frame assignment status/age;
- prevent downstream silent reorder.

Missingness and validity:

- no global blank-to-zero;
- no unlimited measurement forward fill;
- no loss of no-update/invalid/stale provenance;
- no universal invalid-value rule;
- zero/false remains valid unless explicitly rejected;
- invalid defaults are signal-specific.

Signal semantics:

- support FLOAT, INTEGER, BOOLEAN, ENUM, BITFIELD;
- support MEASUREMENT, STATE, EVENT_FLAG, VALIDITY_FLAG, COUNTER,
  IDENTIFIER, FRAME_KEY;
- EVENT_FLAG is never held;
- ENUM/BITFIELD raw code remains traceable.

Raw and derived separation:

- raw decoded update is preserved;
- held/aggregated/fused values do not overwrite raw values;
- derived values have distinct names;
- camera target selection is provisional logic.

Data shape and performance:

- selected signals only;
- chunk-capable read path where practical;
- update stream and FrameCount analysis view remain separate;
- no silent frame reduction for high-rate IMU/SysID use.

For each conflict, report:

- file and evidence;
- current behavior;
- violated contract rule;
- severity: LOW / MEDIUM / HIGH / BLOCKING;
- action: KEEP / EXTRACT_AND_CHANGE / REWRITE / RETIRE.

STEP 6 — Classify reuse

Assign exactly one classification to each important component:

K — KEEP_AS_IS
Pure logic already meets the contract and can be reused without semantic
change.

E — EXTRACT_WITH_MINIMAL_CHANGE
Core logic is useful, but it must be separated from GUI/global state or receive
small interface/error-handling changes.

R — REWRITE_USING_LEGACY_AS_REFERENCE
The behavior or coupling conflicts with the contract, but the legacy code is
useful as a behavioral reference.

X — RETIRE
The responsibility is duplicated, unsafe, or no longer belongs in U001.

Important:

- K is a strict category. Do not use K when missingness, order, provenance,
  data types, or GUI coupling require semantic modification.
- A file is not classified as one unit. Classify its internal components.
- U001 may use P001–P003 as reference without importing their GUI modules.

Map legacy evidence into these approved internal owners:

M11 output writer

For each M01–M13, state:

- legacy source candidates;
- reuse class K/E/R/X;
- contract changes required;
- dependencies;
- first regression check;
- unknowns.

Do not invent a new module unless an unavoidable responsibility is missing.
If proposing one, label it PROPOSED_NOT_APPROVED.

STEP 8 — Define the extraction sequence

Propose an incremental sequence. It must obey these constraints:

- do not begin with a three-GUI merge;
- pure logic before GUI;
- one responsibility and one regression target per step;
- legacy programs remain unchanged;
- each step can be rejected without invalidating completed earlier steps;
- ordering/fill behavior is not copied as an accidental package.

Use this preferred sequence unless source evidence requires a change:

E0 freeze fingerprints and observable legacy behavior;
E1 extract CSV/header detector;
E2 extract DBC parser/decode primitives;
E3 extract inventory/selection;
E4 define configuration/schema objects;
E5 build selected-signal update decoder with SourceRowIndex;
E6 build FrameCount assignment;
E7 build type/validity handling;
E8 build aggregation and bounded hold;
E9 build writers/quality/manifest;
E10 attach GUI controller;
E11 perform legacy-vs-U001 regression and contract validation.

Any changed sequence requires explicit source evidence and reason.

STEP 9 — Identify regression tests

Define source-independent test intentions for:

- little-endian unsigned;
- little-endian signed;
- big-endian unsigned;
- big-endian signed;
- scale and offset;
- missing/incomplete byte rows;
- duplicated source rows;
- repeated FrameCount;
- non-monotonic FrameCount;
- source rows before first FrameCount;
- BOOLEAN false;
- ENUM unknown code;
- EVENT_FLAG pulse;
- signal-specific invalid default;
- bounded hold expiry;
- high-rate multiple updates inside one frame;
- deterministic output naming;
- cancelled run and partial output safety.

Do not generate test code in this gate.

STEP 10 — Produce the final recommendation

Conclude with:

- reusable legacy assets;
- components requiring extraction;
- components requiring rewrite;
- components to retire;
- top three implementation risks;
- unknowns that need runtime/data confirmation;
- whether detailed U001 module design may proceed:
  PROCEED / PROCEED_WITH_CONDITIONS / MORE_INSPECTION / REWRITE_FROM_REFERENCE.

This is a recommendation only. ChatGPT owns Gate approval.
